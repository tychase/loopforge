# LoopForge / Hermes Game Lab — Project Brief

## Core Idea

Build an agentic game-development lab for Vibe Jam 2026.

The system runs on a server with a Hermes-style agent workflow. It studies game design principles, motivation psychology, reward loops, player engagement, and Vibe Jam rules, then generates small playable browser-game experiments.

The final submitted game should be a polished, fast-loading web game. The agentic system can exist behind the scenes as the game-design engine.

## One-Line Pitch

LoopForge is a first-person Vibe Jam arena: collect AI-code shards while the actual contest judges chase, roast, learn, and respawn.

## North-Star Game Direction

The current foundation is a lightweight browser arcade game, but the intended direction is a funny, instantly-playable FPV/open-arena game.

Core fantasy:

- You are trapped inside the Vibe Jam arena.
- You collect shards of AI-generated code, vibe energy, and prototype polish.
- The contest judges are silly character enemies chasing you around the arena.
- The judges can be fought, damaged, defeated, and reset.
- When a judge respawns, it keeps a tiny amount of learned experience from the previous encounter.
- The player earns score from collection, survival, judge defeats, close calls, and eventually public leaderboard placement.

The humor should be affectionate and arcade-like, not mean-spirited. The judges are exaggerated silly arcade mascot sprite billboards: scorecard goblins, deadline demons, vibe inspectors, and roast machines.

## Hackathon Context

Vibe Jam 2026 is an AI game-development competition where 90%+ of the code should be AI-written. The game must run on the web, be free to play, require no login, and load quickly. The deadline is May 1, 2026 at 13:37 UTC. The required Vibe Jam widget script must be included in the game HTML.

## Desired Feel

The player should feel like they are inside an AI-run arcade lab. The game is not just a game; it is an experiment generated by an agentic system.

The experience should be:

- instantly playable
- visually satisfying
- easy to understand in 5 seconds
- replayable
- slightly weird / AI-lab themed
- grounded in real game design and psychology research

## MVP Game Direction

A lightweight first-person Vibe Jam arena.

The player collects cyan AI-code shards, avoids or later fights judge chasers, survives short waves, and receives mutations/upgrades generated by the agent lab. The current implementation can use pseudo-3D canvas projection first; sprites, richer arena art, and true 3D can be layered in after the core mechanics are fun.

Examples:

- faster player movement
- larger pickup radius
- dash / dodge burst
- shard-blast projectile combat as the first fight-back mechanic
- judge health and defeat rewards
- judge respawn with local per-match learned stat or behavior adaptation first, global learning later
- score multiplier for close calls
- random arena mutation
- new hazard
- temporary power-up
- “research hypothesis” shown before each round

## Character/Sprite Direction

Judges should eventually become silly readable sprite characters, not realistic portraits.

Recommended approach:

1. Use silly arcade mascot sprite billboards as the target style.
2. Give every judge a strong silhouette, color, prop, and catchphrase.
3. Use affectionate parody: mascot/boss energy, not personal attacks.
4. Keep sprites low-resolution or hand-drawn-styled so they are funny, fast-loading, and easy to iterate.
5. Animate with cheap sprite-sheet states: idle, chase, attack, hurt, defeated, respawn.
6. Store character data in config: handle, display name, color, sprite paths, barks, health, attack type, learning profile.

## Future Open Arena Direction

The long-term game can become a public open arena where players join, score points, fight judge characters, and compete on a leaderboard. This should be phased carefully so the jam build stays shippable:

1. Local single-player arena first.
2. Local judge health/combat with shard blasts next.
3. Local per-match judge-learning model next.
4. Local/bot fallback must remain available even after network features exist.
5. Server leaderboard and global judge learning after the core loop is fun.
6. Real-time multiplayer/open arena is the target, but only after leaderboard, bot fallback, and anti-abuse basics are stable.

## Important Constraint

Do not attempt to make a fully autonomous AI that writes and deploys completely new games from scratch during the jam.

Instead, build a stable game shell where the agent generates:

- game rules
- parameters
- enemy behavior configs
- reward timing
- upgrade options
- level names
- design hypotheses
- flavor text

The game engine should read these configs and produce playable variations.
